# 我们已经完成了数据模型层的重构，测试覆盖率达到 85%。下一...

## 问题
我们已经完成了数据模型层的重构，测试覆盖率达到 85%。下一步计划重构业务逻辑层。

## 分析
1. **结论**：业务逻辑层重构可按计划启动，但需优先处理数据模型层重构中遗留的 blocker 问题（如 `model_validation` 模块未完全覆盖），以避免后续耦合风险。

2. **依据**：
   - 笔记 #2024-04-15：数据模型层重构完成，测试覆盖率 85%（含单元测试/集成测试）；明确标注 `model_validation` 模块为 *blocker*（当前未覆盖完整边界条件，见 issue #78）。
   - 笔记 #2024-04-20：业务逻辑层依赖数据模型层的接口稳定性，若模型层存在未修复的验证缺陷（如空值处理、时序不一致），将导致业务逻辑层测试失败率上升（参考历史案例：#63 中因模型层校验缺失导致订单创建逻辑异常）。
   - 知识库：《微服务分层重构指南》第 3.2 节强调：“上层重构前必须确保下层接口契约稳定，建议通过契约测试（Contract Test）验证模型层变更对业务层的影响。”

3. **风险与假设**：
   - 风险：若跳过模型层的缺陷修复直接重构业务层，可能引发“测试漂移”——业务层测试通过，但运行时因模型层异常导致业务逻辑错误（参考 #63 的线上事故）。
   - 假设：业务逻辑层重构采用“渐进式替换”策略（即新旧代码并存），可降低回滚成本；但需确认团队对 DDD 领域驱动设计的理解程度（详见笔记 #2024-03-10 的技能评估）。

4. **下一步行动建议**：
   - ✅ **立即行动**：  
     （1）在 `model_validation` 模块补充 3 条边界条件测试用例（参考 issue #78 的待办清单），目标提升至 95% 覆盖率；  
     （2）新增契约测试套件（使用 Pact 或 WireMock），验证模型层接口变更对业务层调用方的影响（参考知识库中的“契约测试模板”）。  
   - 📌 **并行推进**：  
     （1）启动业务逻辑层重构的子任务拆解（如按领域划分：`order` / `payment` 子模块），每模块制定独立验收标准；  
     （2）在 Jira 中创建 `BUSINESS_LOGIC_REFACTORING_01` 任务，关联数据模型层的测试报告（见笔记 #2024-04-15 的附件）。  
   - ⚠️ **阻塞检查**：若 `model_validation` 模块修复延期超过 2 天，则暂停业务层重构，转为“模型层+业务层协同重构”模式（参考笔记 #2024-04-25 的应急方案）。
