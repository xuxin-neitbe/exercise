## 1. 项目概述与目标

### 1.1 项目DNA
**实现用户之间高效、可靠的即时消息传递闭环。**

### 1.2 项目愿景
构建一个轻量级、高性能的点对点即时通讯工具，聚焦核心通讯体验，避免功能膨胀。

### 1.3 核心目标
| 目标类型 | 具体指标 | 衡量标准 |
|----------|----------|----------|
| 用户增长 | V1上线3个月内DAU突破10,000 | 日活跃用户数 |
| 核心体验 | 消息送达率 ≥ 99.9% | 服务端监控 |
| 性能基准 | 消息端到端延迟 ≤ 500ms | P95延迟监控 |

### 1.4 First Action Path
```
用户登录 → 选择/添加联系人 → 发送消息 → 对方实时接收
```

### 3.2 功能详细规格

#### 3.2.1 身份认证模块 [P0]

**功能描述**: 用户通过邮箱+密码或魔法链接完成身份注册与登录。

**数据模型**:
```
User {
  id: UUID (PK, Prisma 生成)
  email: String (Unique)
  name: String
  avatar_url: String
  password: String (可选，用于邮箱密码登录)
  created_at: Timestamp
}
```

> **⚠️ 注意**: User 表数据由 NextAuth.js 管理，支持邮箱密码登录和魔法链接登录。

**状态机**:
```
[未注册] --注册成功--> [注册完成]
[已注册] --登录成功--> [登录成功]
[登录成功] --Token过期--> [需重新验证]
```

**验收标准**:
- 密码长度: 最少8位，包含字母和数字
- 魔法链接有效期: 24小时
- Session 有效期: 30天
- 支持邮箱密码登录和魔法链接登录

#### 3.2.2 点对点消息模块 [P0]

**功能描述**: 用户之间发送文字和图片消息，支持实时送达。

**数据模型**:
```
Message {
  id: UUID (PK)
  content: Text
  sender_id: UUID (FK -> User)
  conversation_id: UUID (FK -> Conversation)
  created_at: Timestamp
  status: Enum[sending, sent, delivered, read, failed]
}

Conversation {
  id: UUID (PK)
  type: Enum[private, group]
  name: String (群聊名称，可为空)
  creator_id: UUID (FK -> User)
  created_at: Timestamp
}
```

> **⚠️ 注意**: 实际实现使用单一 messages 表，通过 conversation_id 关联会话，不区分 C2C/GroupMessage。

**状态机**:
```
MessageStatus:
[creating] --> [sending] --> [sent] --> [delivered] --> [read]
                  |
                  v
               [failed]
```

**验收标准**:
- 文字消息长度上限: 5000字符
- 图片格式支持: JPEG, PNG, WebP
- 图片大小上限: 10MB
- 消息送达延迟: P95 ≤ 500ms

#### 3.2.3 群聊模块 [P1]

**功能描述**: 支持创建群组进行多人沟通，使用 Member 关联表实现。

**数据模型**:
```
Conversation (群聊) {
  id: UUID (PK)
  type: 'group'
  name: String (群名称)
  creator_id: UUID (FK -> User)
  created_at: Timestamp
}

Member {
  id: UUID (PK)
  user_id: UUID (FK -> User)
  conversation_id: UUID (FK -> Conversation)
  role: Enum[owner, admin, member]
  is_muted: Boolean (是否被禁言)
  muted_until: Timestamp (禁言截止时间，可为null表示永久禁言)
  muted_by: UUID (FK -> User, 禁言操作者)
  created_at: Timestamp
}
```

> **⚠️ 注意**: 实际实现使用 Conversation + Member 表，不使用独立的 Group/GroupMember 表。

**验收标准**:
- 群成员上限: 50人（V1硬限制）
- 群消息广播延迟: P95 ≤ 1s
- 支持群主转让、成员管理

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 消息端到端延迟 | P95 ≤ 500ms | 客户端埋点 |
| API响应时间 | P95 ≤ 200ms | 服务端监控 |
| 并发连接数 | ≥ 100,000 | 压力测试 |
| 图片上传速度 | 1MB/s (4G环境) | 客户端监控 |

### 4.2 可用性需求

| 指标 | 目标值 |
|------|--------|
| 服务可用性 | ≥ 99.9% (月度) |
| 计划内停机 | ≤ 4小时/月 |
| 故障恢复时间 | ≤ 30分钟 |

### 4.3 安全需求

| 安全项 | 要求 |
|--------|------|
| 传输加密 | TLS 1.3 |
| 密码存储 | 不存储密码，仅验证码认证 |
| 敏感数据 | 手机号脱敏显示 |
| 会话管理 | Token有效期7天，支持主动注销 |

### 4.4 合规需求

| 合规项 | 要求 |
|--------|------|
| 数据存储 | 用户数据存储于中国大陆服务器 |
| 内容审核 | 接入第三方内容安全API |
| 用户协议 | 注册前强制展示并确认 |

## 7. 接口设计概要

## 8. 风险点识别

> **[首席风险审计官 - 2026-02-13]**

### 8.1 技术风险

| 风险项 | 等级 | 影响 | 缓解措施 |
|--------|------|------|----------|
| WebSocket连接管理 | 🟠 严重 | 10万并发连接需要精细的连接管理和心跳机制，连接泄漏、内存溢出、消息丢失风险 | 1. 实现连接池管理和自动重连机制；2. 心跳检测和僵尸连接清理；3. 消息ACK机制确保送达 |
| 消息顺序性与幂等性 | 🟠 严重 | 群聊场景下消息广播的顺序性和去重是典型难题，消息乱序、重复投递 | 1. 使用分布式序列号生成器；2. 客户端消息去重缓存；3. 消息ID作为幂等键 |
| 微服务架构复杂度 | 🟡 中等 | 服务拆分过细（6个服务）导致运维复杂度高，故障排查困难 | MVP阶段建议采用单体架构或合并服务，降低运维负担 |
| 数据库性能瓶颈 | 🟡 中等 | 单表消息量增长后查询性能急剧下降，历史消息查询超时 | 1. 按会话ID分片；2. 冷热数据分离（超3个月归档）；3. 预先设计分片策略 |
| 弱网环境适配 | 🟡 中等 | 移动端网络不稳定，消息发送失败率高，用户流失 | 1. 本地消息队列缓存；2. 指数退避重试机制；3. 离线消息同步机制 |
| 第三方服务单点故障 | 🟢 轻微 | 短信服务、对象存储、内容审核API均为第三方，存在服务中断风险 | 多服务商备份方案，核心功能降级策略 |

### 8.2 合规风险

| 风险项 | 等级 | 影响 | 缓解措施 |
|--------|------|------|----------|
| 增值电信业务许可证 | 🔴 致命 | 即时通讯属于B25类增值电信业务，无证运营面临罚款、关停、刑事责任 | 上线前必须完成ICP备案、申请增值电信业务经营许可证，或接入具备资质的第三方IM SDK |
| 内容审核义务 | 🔴 致命 | 《网络安全法》第47条要求平台审核用户信息，违规内容传播导致下架、吊销许可 | 1. 消息发送接口接入实时内容审核（文本+图片）；2. 建立敏感词库；3. 保留审核日志至少6个月 |
| 实名制认证 | 🟠 严重 | 《网络安全法》第24条要求即时通讯实施实名制，无法通过应用商店审核 | 手机号验证码登录已满足实名制要求，需确保手机号真实有效，评估是否限制虚拟运营商号码 |
| 个人信息保护法合规 | 🟠 严重 | 手机号属敏感个人信息，需遵循"最小必要"、"明示同意"原则，最高罚款5000万或营收5% | 1. 隐私政策注册前强制展示并确认；2. 明确告知手机号用途；3. 提供账号注销功能；4. 数据删除可追溯 |
| 数据本地化存储 | 🟢 轻微 | 《数据安全法》要求关键数据境内存储 | PRD已明确数据存储于中国大陆服务器，符合要求 |
| 用户协议与隐私政策 | 🟡 中等 | 格式条款需符合《消费者权益保护法》，否则可能被认定无效 | 法务审核用户协议和隐私政策，确保条款合法有效 |

### 8.3 商业风险

| 风险项 | 等级 | 影响 | 缓解措施 |
|--------|------|------|----------|
| 市场竞争格局 | 🔴 致命 | 微信、QQ等巨头已占据绝对主导，用户迁移成本极高，产品定位需差异化 | 明确差异化定位（如隐私保护、极简体验），避免正面竞争，聚焦细分场景 |
| 变现路径缺失 | 🟠 严重 | PRD未明确商业模式，免费运营将导致持续亏损，无法支撑长期运营 | 1. MVP阶段验证用户需求；2. 探索增值服务（如高级功能订阅）；3. B端私有化部署 |
| 用户冷启动困境 | 🟠 严重 | 新用户无社交关系链，产品价值无法体现，留存率极低 | 1. 通讯录导入降低冷启动成本；2. 邀请奖励机制；3. 种子用户运营 |
| 用户获取成本高 | 🟡 中等 | IM产品获客成本高，且用户粘性依赖社交关系链 | 聚焦垂直场景（如小团队协作），降低获客成本 |

### 8.4 运营风险

| 风险项 | 等级 | 影响 | 缓解措施 |
|--------|------|------|----------|
| 短信验证码费用失控 | 🟠 严重 | 恶意刷取可导致日费用失控，一夜消耗数万元 | 1. 图形验证码/滑块验证前置；2. 单IP/单设备每日发送上限；3. 接入短信服务商风控API；4. 账户余额预警和自动熔断 |
| 内容审核成本 | 🟡 中等 | 第三方内容审核按次计费，日均10万条消息月费用约3000-5000元 | 1. 建立本地敏感词库初筛；2. 仅对通过初筛的内容调用API；3. 用户信誉分级，低风险用户抽样审核 |
| 对象存储与CDN成本 | 🟡 中等 | 图片存储按量和流量计费，DAU达1万时月费用可能超万元 | 1. 图片压缩和WebP格式转换；2. 图片过期自动清理；3. CDN流量包预购 |
| 基础设施成本 | 🟡 中等 | 微服务架构多节点部署，零用户时月成本也在3000-5000元 | MVP阶段采用单体架构或容器化部署，降低运维成本 |
| 内容审核漏审风险 | 🟠 严重 | 用户规模扩大后审核压力增加，漏审违规内容导致法律风险 | 1. 多级审核机制；2. 用户举报通道；3. 人工复审流程 |

### 8.5 风险审计结论

**审计结论: 建议项目暂缓推进，待致命风险项解决后再启动开发。**

**核心问题清单（必须解决）:**

1. **[阻断项]** 增值电信业务许可证（B25类）未取得，无证运营属于违法行为。建议：
   - 方案A: 申请增值电信业务经营许可证（周期约3-6个月）
   - 方案B: 接入具备资质的第三方IM SDK（如融云、环信），降低合规门槛

2. **[阻断项]** 内容审核能力未明确技术方案，仅提及"接入第三方API"不足以满足合规要求。建议：
   - 明确审核服务商（阿里云内容安全/腾讯云天御/网易易盾）
   - 设计审核流程：发送前审核 vs 发送后审核
   - 评估审核延迟对用户体验的影响

3. **[阻断项]** 商业模式缺失，无明确变现路径。建议：
   - MVP阶段前明确商业模式假设
   - 评估单位经济模型（CAC vs LTV）
   - 设定止损线（如6个月内无法实现正向现金流则终止项目）

**条件性通过建议:**

若上述阻断项得到有效解决，项目可进入MVP开发阶段，但需满足以下前置条件：
- 完成隐私政策和用户协议的法务审核
- 短信验证码防刷机制上线前必须就位
- 内容审核API接入并完成联调测试

## 9. MVP边界建议

### 9.1 核心生存链路 (Golden Loop)

### 9.2 MVP核心目标

**验证假设**：用户愿意使用这个IM工具进行点对点文本沟通。

### 9.3 MVP功能清单

| 功能 | 是否纳入 | 理由 |

| 点对点文本消息 | ✅ | IM核心DNA，唯一价值交换 |

| 群聊功能 | ❌ | 引入多状态变更和权限体系，超出V1 |

| 消息撤回 | ❌ | 引入补偿机制，超出V1 |
| 通讯录导入 | ❌ | 非核心链路功能 |

| 历史消息查询 | ❌ | 本地存储即可满足MVP需求 |

#### 9.3.1 简化版好友系统规格

**设计原则**：最小化社交复杂度，满足"找到人、发消息"的核心需求。

**功能范围**：
| 子功能 | 是否纳入 | 说明 |
|--------|----------|------|
| 通过手机号搜索用户 | ✅ | 核心查找方式 |
| 通过用户ID搜索用户 | ✅ | 备用查找方式 |
| 直接添加好友（无需审批） | ✅ | 简化流程，降低社交门槛 |
| 好友列表展示 | ✅ | 选择聊天对象的入口 |
| 删除好友 | ✅ | 基础管理能力 |
| 好友申请/审批流程 | ❌ | 延后V1.2，V1采用"直接添加"模式 |
| 好友备注名 | ❌ | 延后V1.2 |
| 好友分组 | ❌ | 延后V1.3 |

**数据模型**：
```
Friendship {
  friendship_id: UUID (PK)
  user_id: UUID (FK -> User)
  friend_id: UUID (FK -> User)
  created_at: Timestamp
  PK: (user_id, friend_id)  // 单向关系，A添加B后A可发消息给B
}
```

**核心链路修正**：
